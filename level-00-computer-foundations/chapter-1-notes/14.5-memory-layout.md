# Lesson 14.5

Memory Layout (Conceptual)

Suppose your computer has:

16 GB RAM

When you power on your computer...

Is all 16 GB empty and waiting for Python?

No.

Not even close.

What Happens When You Turn On the Computer?

Let's follow the boot process at a high level.

## Step 1: Power On

Electricity reaches the motherboard.

The CPU wakes up.

At this moment:

RAM is essentially empty.
No operating system is running.
No applications exist in memory.

The CPU has one question:

"What instruction should I execute first?"

It doesn't magically know.

## Step 2: Firmware Takes Over

The CPU starts executing firmware stored in ROM/flash memory (modern systems use UEFI; older ones used BIOS).

The firmware:

checks the CPU,
checks the RAM,
checks storage devices,
finds a bootable drive,
loads the operating system.

So the first program the CPU runs is not Windows, Linux, or macOS—it's firmware.

## Step 3: The Operating System Loads

The firmware copies the operating system's kernel from the SSD into RAM.

Now the CPU begins executing the kernel.

The kernel is the core of the operating system.

From this point onward, the OS is in control.

RAM Is No Longer Empty

After booting, RAM might conceptually look like this:

    +--------------------------------------+
    | Operating System Kernel               |
    +--------------------------------------+
    | Device Drivers                        |
    +--------------------------------------+
    | Background Services                   |
    +--------------------------------------+
    | Free Memory                           |
    |                                       |
    |                                       |
    |                                       |
    +--------------------------------------+

Notice something.

Before you even open a browser or a terminal, a significant portion of RAM is already in use.

Opening Python

Now you type:

python app.py

The OS creates a new process and loads the Python interpreter into RAM.

Conceptually:

    +--------------------------------------+
    | Operating System                      |
    +--------------------------------------+
    | Drivers                               |
    +--------------------------------------+
    | Background Services                   |
    +--------------------------------------+
    | python.exe Process                    |
    |                                       |
    | Code                                  |
    | Data                                  |
    | Stack                                 |
    | Heap                                  |
    |                                       |
    +--------------------------------------+
    | Free Memory                           |
    +--------------------------------------+

We haven't defined stack and heap yet.

For now, just notice that the process gets its own area.

Opening Chrome

Now you open Chrome.

RAM now looks more like:

    +--------------------------------------+
    | Operating System                      |
    +--------------------------------------+
    | Drivers                               |
    +--------------------------------------+
    | Background Services                   |
    +--------------------------------------+
    | Chrome Process                        |
    +--------------------------------------+
    | Python Process                        |
    +--------------------------------------+
    | Free Memory                           |
    +--------------------------------------+

Each running program gets its own process.

Each process gets its own memory space.

Why Is This Important?

Imagine there were no separation.

Suppose Chrome stores:

Username
Password
Cookies

Python accidentally writes over those bytes.

Chrome crashes.

Or worse...

It steals passwords.

That would be a disaster.

Process Isolation

Modern operating systems isolate processes.

Think of an apartment building.

Apartment 1 → Chrome

Apartment 2 → Python

Apartment 3 → Spotify

Apartment 4 → VS Code

Each apartment has:

its own furniture,
its own belongings,
its own keys.

The people in Apartment 2 cannot walk into Apartment 1 whenever they want.

Processes work similarly.

Wait...

If RAM is just one physical chip...

How can each process think it owns memory?

Excellent question.

The answer is virtual memory.

Virtual Memory (Conceptual)

This topic becomes very deep in operating systems, but for now we only need the intuition.

Each process is given the illusion that it has its own private memory.

Python might think:

My first byte is at address:

0x1000

Chrome might also think:

My first byte is at address:

0x1000

How can that be?

Because those are virtual addresses, not physical RAM locations.

The operating system and the CPU's Memory Management Unit (MMU) translate virtual addresses into physical RAM addresses behind the scenes.

This gives each process a private view of memory.

Why Is This Amazing?

Imagine two terminals both running:

python app.py

Both processes may internally use the same virtual address for a variable, yet they don't interfere with each other because the OS maps those virtual addresses to different physical locations.

This is why two copies of the same program can run independently.

Where Does app.py Live?

This is a subtle but important point.

Before you run it:

app.py

lives on the SSD.

After you run it:

the file still exists on the SSD,
parts of its contents are loaded into RAM,
Python parses and compiles it,
execution happens in RAM.

The original file is not "consumed."

Why Doesn't a Crash Destroy the File?

Suppose your program crashes.

What gets destroyed?

The process.

Not the source file.

The file on the SSD remains untouched unless your program explicitly writes to it.

This distinction between persistent storage and temporary execution state is fundamental.

Free Memory

Not all RAM is used.

There must always be free space for:

opening new applications,
allocating new objects,
buffering files,
network data,
AI model tensors,
browser tabs.

When free RAM becomes scarce, the operating system may use disk-based virtual memory (swap/page file), which is much slower than RAM.

## AI Connection

Imagine loading a 20 GB language model.

If your computer has:

16 GB RAM

the model cannot fit entirely in memory.

The OS must constantly move data between RAM and storage.

This is called paging or swapping, and it's one reason large AI models can become extremely slow on machines with insufficient RAM.

Common Misconceptions
 "RAM belongs to my program."

No.

RAM belongs to the operating system.

The OS allocates portions of it to processes.

"Programs run from the SSD."

No.

Programs are stored on the SSD but execute from RAM.

"If two programs use the same address, they overwrite each other."

Not with virtual memory.

The same virtual address in different processes can map to different physical memory.

## Mental Model

Think of RAM as a hotel managed by the operating system.

The OS is the hotel manager.
Each process is a guest.
Each guest gets a private room.
Guests cannot enter each other's rooms without permission.
When a guest checks out (process ends), the room becomes available for someone else.


## Key Insights

- RAM is managed by the operating system.
- Every running application is a separate process.
- Each process gets its own isolated virtual memory space.
- Programs are stored on SSD but execute from RAM.
- Virtual memory allows multiple processes to run safely without interfering with one another.
